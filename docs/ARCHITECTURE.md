## The problem this replaces

A "legacy static ETL pipeline" in a supply chain context typically looks like this: a scheduled script runs every night, pulls the same data from the same source, applies the same fixed transformation, and writes a report. It has no concept of "this situation looks unusually risky, let me double check" — it just runs the same steps regardless of what the data actually says.

That's a real limitation. If a supplier in a disruption zone has unusually low reliability, a static pipeline doesn't know to dig deeper. It just produces its usual output.

**What we built instead** is a system where the *path through the workflow itself changes* based on what's discovered along the way:

- If incoming data fails validation, the workflow stops immediately and produces an error report instead of silently working with bad data.
- If a stress-test scenario looks severe enough that the first simulation pass might not be statistically reliable, the system automatically re-runs the simulation with more iterations before trusting the result.
- The final report is generated by an LLM that reasons over the *structured results*, not a fixed report template with blanks filled in.

This is what "agentic" means in practice: the system makes decisions about its own next steps, not just data transformations.

## High-level architecture

```
Client (curl / Swagger UI / any HTTP client)
        |
        v
FastAPI layer            app/api/routes.py
        |
        v
Orchestration service     app/services/orchestration_service.py
        |
        v
LangGraph workflow         app/agents/graph.py  (5 agent nodes, conditional edges)
        |
        +--> Simulation engine   app/services/simulation_engine.py   (Monte Carlo / NumPy)
        |
        +--> LLM service         app/services/llm_service.py         (AWS Bedrock or Mock)
        |
        v
data/reports/*.md   (generated report saved to disk)
```

See `diagrams/system_architecture.svg` for the visual version.

Each layer has exactly one job:

| Layer | Responsibility | Does NOT do |
|---|---|---|
| FastAPI routes | Parse/validate HTTP input, call the orchestration service, shape the HTTP response | Business logic, agent logic |
| Orchestration service | Build the initial state, invoke the compiled graph, expose results cleanly | HTTP concerns, agent logic |
| LangGraph workflow | Decide *which agent runs next* based on what previous agents found | HTTP, raw I/O, LLM calls |
| Agents (5 files) | One specific job each (ingest, assess risk, simulate, forecast, report) | Deciding graph routing themselves (they just *report* their preferred next step) |
| Services (simulation engine, LLM client, data loader) | Pure, testable business/math logic | Anything about LangGraph or HTTP |

This is a standard layered architecture, and the reason it matters for an interview: you can explain *any single file's job* in one sentence, which is exactly what a code reviewer or interviewer is listening for.

## Why LangGraph

LangChain alone gives you **chains**: A -> B -> C, always in that order. That's enough for many tasks, but not for this one, because we need:

1. **Conditional branching** — "if the data is bad, skip straight to an error report instead of running the rest of the pipeline."
2. **Loops** — "if the simulation result doesn't look trustworthy given the risk level, run it again with more iterations, then re-check — possibly several times — before continuing."
3. **A shared, explicit state object** that every step can read from and write to, with a clear contract for how concurrent writes get merged (LangGraph's `Annotated[list, operator.add]` pattern, used here for the audit log).

LangGraph models the workflow as an actual **graph**: nodes (agents) connected by edges, where edges can be conditional functions evaluated at runtime. That's precisely the data structure this problem needs. See `app/agents/graph.py` — every edge in that file is a direct line you can point to in an interview and say "this is the part a static pipeline literally cannot express."

## The agent graph in detail

See `diagrams/langgraph_workflow.svg` for the visual.

### 1. Data Ingestion Agent (`app/agents/data_ingestion_agent.py`)

Loads supplier and inventory data (currently from local JSON files in `data/sample/`, deliberately structured so swapping this for a real database or API call later is a one-file change — see `app/services/data_loader.py`). Validates referential integrity (every inventory item must point to a real supplier). If validation fails, it sets `next_agent = "end"` and the graph terminates immediately with an error report instead of wasting compute on a doomed simulation.

### 2. Risk Assessment Agent (`app/agents/risk_assessment_agent.py`)

Computes a composite risk score (0.0 to 1.0) from three deterministic factors:
- The requested scenario severity (given directly in the API request)
- What fraction of suppliers are in the affected region
- What fraction of SKUs have less buffer stock than the disruption is expected to last

This is intentionally **rule-based, not an LLM call**. Risk scoring needs to be explainable and reproducible — if a VP of Operations asks "why did the system say this was HIGH risk," the answer needs to be a formula you can show them, not "the AI decided." This is a deliberate, defensible engineering choice, and a good thing to say explicitly in an interview: *not every agent in an agentic system needs to call an LLM — agents are about decision-making and orchestration, the LLM is just one tool some agents use.*

### 3. Simulation Agent (`app/agents/simulation_agent.py`) — the key agentic behavior

Runs a Monte Carlo stress test (via the simulation engine service) and then makes a decision:

```
if risk_score >= ESCALATION_THRESHOLD
   and stockout_probability > 0.5
   and we haven't hit the max retry limit yet:
       -> re-run the simulation with MORE iterations (better statistical confidence)
       -> loop back to this same agent node
else:
       -> proceed to the Forecasting Agent
```

This loop-back edge (`simulation -> simulation`) is registered explicitly in `app/agents/graph.py` and is the single clearest piece of evidence that this is a *graph*, not a chain. Each retry increases the Monte Carlo run count (1000, then 2000, then 3000...), so a retry isn't just "try again and hope" — it's "get a statistically tighter answer before trusting it."

### 4. Forecasting Agent (`app/agents/forecasting_agent.py`)

Takes the simulation's expected revenue impact and spreads it across a 12-month forecast horizon, modeling the disruption window and a recovery taper, to produce baseline vs. stressed year-over-year growth numbers. This is formula-driven (not a trained ML model) by design — see [Design decisions](#design-decisions-and-trade-offs) for why, and how you'd evolve it later.

### 5. Report Generation Agent (`app/agents/report_generation_agent.py`)

The only agent that calls an LLM. It builds a prompt containing a structured JSON block with every number produced by the previous four agents, and instructs the model: *narrate this, do not invent or alter any numbers*. This guardrail — numbers come from code, prose comes from the model — is standard practice for any "AI writes a business report" feature and is worth stating explicitly in an interview.

## Shared state design

LangGraph passes a single state object between every node. We define it in `app/models/state.py` as a `TypedDict` (LangGraph's recommended pattern, because LangGraph needs to merge partial dict updates between nodes — a Pydantic model would fight that).

Key design choices:

- **`Annotated[list, operator.add]`** on `audit_log` and `errors` — this tells LangGraph "when multiple nodes write to this field, concatenate the lists instead of overwriting." This is how we get a full, ordered audit trail of every decision every agent made over the whole run, without any agent needing to know about any other agent's logging.
- **`next_agent` field** — every agent writes its preferred next step into the state. A single `route_next()` function (in `app/agents/graph.py`) reads this field and tells LangGraph where to go. This keeps all routing logic in one place instead of scattering conditional graph-wiring across every agent file.
- **Field names avoid collisions with node names** — LangGraph forbids a state field and a node from sharing a name (we hit this directly during development: `risk_assessment` as both a node name and a state key raised a `ValueError`). Fields are suffixed `_result` where needed (`risk_assessment_result`, `forecast_result`) to avoid this.

## LLM abstraction: Bedrock with mock fallback

See `app/services/llm_service.py`.

The Report Generation Agent needs an LLM. In production that's AWS Bedrock. But requiring AWS credentials just to run or demo the project is an unnecessary barrier — and an interviewer reading this code shouldn't need an AWS account to see it work.

So the file defines one abstract interface:

```python
class LLMClient(ABC):
    def invoke(self, system_prompt: str, user_prompt: str) -> str: ...
```

with two implementations — `BedrockLLMClient` (a real `boto3` `bedrock-runtime` call) and `MockLLMClient` (an offline, deterministic template engine that reads the same structured JSON block out of the prompt and renders a genuinely readable markdown report from it, not a "lorem ipsum" placeholder).

A single factory function, `get_llm_client()`, decides which one to return based on whether AWS credentials are present in the environment (`app/core/config.py`'s `bedrock_available` property). **No other file in the codebase needs to know or care which implementation it got** — this is the Strategy design pattern, and it's exactly how you'd want to structure *any* swappable external dependency (LLM provider, payment provider, email provider) in a real production system, not just for offline-demo convenience.

If you later add real AWS credentials to `.env`, the exact same code path automatically switches to live Bedrock calls — nothing else in the system changes.

## The simulation engine

See `app/services/simulation_engine.py`.

For each Monte Carlo run, for each affected inventory item:

1. Draw a random delay (gamma-distributed, scaled by the supplier's reliability score — less reliable suppliers get wider, larger delay distributions).
2. Compare that delay against the item's "buffer days" (how long current stock lasts above the reorder point, given average demand).
3. If the delay exceeds the buffer, record a stockout, the resulting lost-sales revenue impact, and an estimated recovery time.

Across all runs, we compute: stockout probability, expected revenue impact, the 90th-percentile ("P90", a standard risk metric for tail scenarios) impact, and the absolute worst case.

This is a deliberately **simple, explainable simulation model** — every number traces back to a stated assumption (see the comments in the file). That's a strength for an interview: you can walk through the math from first principles instead of saying "it's a black box."

## The FastAPI layer

See `app/api/routes.py` and `app/main.py`.

Three endpoints:

- `POST /api/v1/stress-test` — triggers the full agent workflow, returns the complete result (risk assessment, simulation summary, forecast, report, audit log) as JSON.
- `GET /api/v1/reports/{slug}` and `GET /api/v1/reports` — fetch previously generated reports from disk.
- `GET /api/v1/health` — liveness probe, also reports whether the system is in "live" (real Bedrock) or "mock" mode.

Route handlers are intentionally thin — they parse input (FastAPI + Pydantic do this automatically from type hints), call exactly one service function, and shape the response. All business logic lives in the service/agent layers, which keeps the HTTP layer trivially easy to test (see `tests/test_api.py`, which uses FastAPI's `TestClient` to call these endpoints in-process without starting a real server).

## Error handling and data validation

Two distinct failure modes are handled differently, on purpose:

1. **Bad input from the client** (e.g. an invalid `disruption_type`, or `severity` outside 0-1): caught automatically by Pydantic at the FastAPI boundary, returns HTTP 422 before any agent even runs.
2. **Bad data discovered during ingestion** (e.g. an inventory record pointing at a supplier ID that doesn't exist): caught by the Data Ingestion Agent's validation step, which sets `data_quality_ok = False` and routes the graph straight to an error exit. The API layer detects this case and returns a 422 with the specific list of problems found — a clear signal this is a *data* problem, not a *server* bug (which would be a 500).

## Design decisions and trade-offs

Worth being upfront about these in an interview — showing you understand the trade-offs you made is more impressive than pretending the design is perfect.

- **Rule-based risk scoring instead of an LLM call**: more explainable and reproducible; the trade-off is it can't pick up on nuanced patterns an LLM might notice in free-text supplier notes. A natural "v2" extension: have the Risk Assessment Agent call an LLM to read supplier risk disclosures or news sentiment, and *blend* that with the deterministic score.
- **Formula-driven YoY forecasting instead of a trained time-series model**: fast to build, fully explainable, and appropriate for a v1. A real production evolution would swap this for a trained forecasting model (Prophet, a gradient-boosted model, or an LSTM) trained on historical revenue data — the agent's *interface* (read simulation summary, write a forecast) wouldn't need to change at all, which is exactly the point of keeping these concerns cleanly separated.
- **JSON files instead of a real database**: appropriate for a demo project; `app/services/data_loader.py` is the only file that would need to change to point at Postgres, a data warehouse, or a live ERP/SCM API instead.
- **Synchronous graph execution**: the current API call blocks until the whole workflow finishes (typically well under a second with the mock LLM, a few seconds with a real Bedrock call). For a production system handling many concurrent long stress tests, you'd add a task queue (Celery, or AWS SQS + a worker) and make the endpoint return a `run_id` immediately, with a separate endpoint to poll for results. The orchestration service is already structured so adding that later doesn't require touching the agent logic at all.
